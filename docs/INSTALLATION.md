# Установка и обновление

## Чистая установка

1. Скачайте из нужного [GitHub Release](https://github.com/elftorg/keyrights/releases)
   два assets: `drdroid.keyrights-<версия>.tar.gz` и `SHA256SUMS`. Архивы
   `Source code` для установки не подходят.
2. Проверьте архив:

   ```bash
   sha256sum -c SHA256SUMS
   ```

3. Распакуйте его во временный каталог и скопируйте только папку модуля:

   ```bash
   mkdir keyrights-release
   tar -xzf drdroid.keyrights-<версия>.tar.gz -C keyrights-release/
   cp -a keyrights-release/drdroid.keyrights /bitrix/modules/
   ```

   README, инструкции, changelog и LICENSE находятся в корне архива рядом с
   `drdroid.keyrights/`; они не копируются внутрь каталога модуля. Результатом
   должен быть путь `/bitrix/modules/drdroid.keyrights/`.
4. В административной части Bitrix24 откройте Marketplace → Установленные
   решения (список модулей), выберите `drdroid.keyrights` и нажмите «Установить».
   Для новой установки сохраните автоматически созданную ключ-фразу или
   задайте свою длиной не менее 16 символов. Пустое поле безопасно заполняется
   серверным генератором при отправке формы.

Перед чистой установкой рекомендуется передать `KEYRIGHTS_SERVER_KEY` длиной
не менее 32 символов в окружение PHP-FPM/Apache. Тогда серверный ключ не будет
записан в таблицу настроек Bitrix. Не меняйте переменную после появления
зашифрованных записей без отдельной процедуры перешифрования.

Без переменной окружения ключи шифруются штатным `crypto.crypto_key` Bitrix из
`.settings.php`. Проверьте наличие этого ключа и включите `.settings.php` в
защищённую резервную копию. Открытое хранение используется только как режим
совместимости, если системный ключ Bitrix отсутствует.

Перед установкой выполняется server-side preflight: проверяются тип СУБД и
тестовая запись во временную таблицу, версии PHP и Bitrix, OpenSSL и CSPRNG,
модули «Информационные блоки» и «Управление структурой», а также доступность
всех копируемых исходных файлов. Отсутствующие `/keyrights/`, каталоги
компонента, `urlrewrite.php` и `.left.menu.php` не обязаны существовать:
установщик проверяет возможность их создания записью, чтением и удалением
уникального временного объекта в ближайшем существующем родительском каталоге.
Если целевой файл уже существует, он открывается на запись без изменения
содержимого. Изменение прав проверяется только на созданных временных объектах;
ошибка `chmod` показывается как предупреждение и не блокирует установку. Все
ошибки содержат точный путь и неуспешную операцию. После успешной проверки
создаются служебные таблицы, регистрируются обработчики и копируется компонент.

Таблицы старой версии не являются условием установки: если `sib_kr_item` и
`sib_kr_right` отсутствуют, установщик создаёт новую схему и продолжает работу.
Если существуют обе legacy-таблицы, установщик в транзакции копирует и
сопоставляет записи с `dr_kr_item` и `dr_kr_right`. Миграцию можно безопасно
повторить: уже перенесённые записи не дублируются. При наличии только одной из
двух таблиц установка останавливается, чтобы не скрыть повреждённую или
неполную схему. Старые таблицы не изменяются и не удаляются ни установкой, ни
удалением новой версии. После обновления сравните количество и доступность
записей, прежде чем принимать отдельное решение об архивировании старых таблиц.

Все ресурсы установщика локальны. Доступ в интернет и PHP cURL не требуются.
Серверный ключ, создаваемый установщиком при отсутствии переменной окружения,
генерируется через `random_bytes()`.

## Перед обновлением

Перед изменением production-окружения:

1. сделайте резервную копию файлов `/bitrix/modules/drdroid.keyrights/` и
   `/bitrix/components/drdroid/keyrights/`;
2. сделайте резервную копию базы данных;
3. проверьте наличие OpenSSL и доступность Composer-зависимостей;
4. протестируйте установку в staging.

## Удаление

Удаление выполняется штатным Bitrix-установщиком. Перед удалением сохраните
резервную копию: операция удаляет созданные модулем таблицы и разделы данных.
