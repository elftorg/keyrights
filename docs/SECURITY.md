# Безопасность

- операции записи проходят через POST и CSRF-фильтр;
- API проверяет авторизацию и права владельца/администратора;
- обновление прав проверяет принадлежность записи целевому разделу;
- `rights/remove` не принимает произвольный объект без проверки доступа;
- дерево разделов фильтруется на сервере; скрытые ветки не раскрывают названия
  и списки ACL, а справочник сотрудников доступен только управляющим правами;
- перемещения проверяют права одновременно на исходный и целевой разделы;
- строки CSV, похожие на формулы, нейтрализуются перед экспортом;
- запросы доступа ограничены по частоте, а BBCode в названиях экранируется;
- импорт доступен только администратору, ограничен 5000 строками и 10 МиБ на
  HTTP-запрос; промежуточное состояние шифруется AES-256-GCM серверным ключом,
  ограничено 8 МиБ, живёт 30 минут и привязано к пользователю;
- JSON API не смешивает параметры запроса с Cookie: JSON разбирается только
  при соответствующем Content-Type, а fallback выбирается между GET и POST по
  HTTP-методу; ответы запрещают MIME-sniffing, framing и передачу referrer;
- новые шифротексты имеют версию и аутентифицированное шифрование
  AES-256-GCM;
- новые клиентские шифротексты используют PBKDF2-HMAC-SHA256 (210 000
  итераций), а nonce и генерируемые пароли получают случайность только через
  `crypto.getRandomValues()`; прежний формат остаётся доступен только для
  совместимого чтения и переводится в новый при сохранении;
- legacy-дешифратор сохраняется для чтения ранее созданных записей.
- копирование чувствительных значений выполняется только в браузере через
  Clipboard API или локальный `document.execCommand`; Flash, ZeroClipboard и
  отправка содержимого на внешний сервер не используются.
- отчёты об ошибках автоматически никуда не отправляются. Предложения и
  сообщения об ошибках публикуются вручную через
  [GitHub Issues](https://github.com/elftorg/keyrights/issues).

Секреты, ключевые фразы и production-конфигурация не должны попадать в
репозиторий, логи CI или релизные заметки.

Для новых установок задайте серверный ключ через `KEYRIGHTS_SERVER_KEY`
(не менее 32 символов) до установки. Выбранный источник фиксируется в
настройках модуля; произвольная смена ключа повредит возможность расшифровки.
При отсутствии переменной модуль защищает ключи через системный
`crypto.crypto_key` Bitrix; открытый fallback оставлен только для старых
конфигураций без штатной криптографии.
Клиентская ключевая фраза необходима браузеру для локальной расшифровки, но
больше не возвращается отдельным API, выдаётся только пользователю с доступом к
хранилищу и удаляется из глобального объекта конфигурации сразу после
инициализации frontend-кода. Это не устраняет архитектурный риск XSS или
вредоносного расширения браузера: доступный пользователю plaintext и ключ могут
быть прочитаны кодом, выполняющимся в его сессии. Поэтому портал должен работать
только по HTTPS, а сторонние скрипты и расширения браузера следует ограничивать.
