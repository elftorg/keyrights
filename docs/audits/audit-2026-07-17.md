## Анализ безопасности

### 🔴 Критический (Критический) — 0

Критических уязвимостей не обнаружено. Модуль демонстрирует зрелый подход к безопасности.

### 🟠 Высокий (Высокий) — 1

#### 1. Потенциальный CSRF на endpoint `api.php` при GET-запросах

**Файл:** `install/frontend/api.php:13`
**Код:**
```php
$route = isset($_REQUEST['action']) ? trim((string)$_REQUEST['action'], '/') : '';
```

**Проблема:** API endpoint `/keyrights/api.php` принимает параметр `action` через `$_REQUEST`, что допускает GET-запросы. Несмотря на то, что сам контроллер проверяет CSRF для write-действий, точка входа `api.php` за пределами Bitrix-компонента может быть вызвана без `bitrix_sessid()`.

**Рекомендация:** Добавить проверку `check_bitrix_sessid()` или ограничить `api.php` только POST-запросами на уровне входа.

#### 2. Утечка путей файловой системы через `$_SERVER['DOCUMENT_ROOT']`

**Файл:** `classes/general/ckeyrights.php:202-204`
```php
$staticPath = '/bitrix/components/drdroid/keyrights/static';
$staticRoot = $_SERVER['DOCUMENT_ROOT'] . $staticPath;
$assetVersion = max(
    (int)@filemtime($staticRoot . '/css/style.css'),
    ...
```

**Проблема:** При ошибках PHP путь `DOCUMENT_ROOT` может быть раскрыт. Используется подавление ошибок (`@`), что является допустимым, но не идеальным решением.

### 🟡 Средний (Средний) — 2

#### 1. Сериализованные данные в PHP-сессии

**Файл:** `lib/model/import.php:107`
```php
$_SESSION[self::SESSION_KEY] = $state;
```

**Проблема:** Импорт хранит до 8 МБ сериализованных данных в сессии PHP. При недостаточной изоляции сессий на виртуальном хостинге возможна утечка данных паролей (сериализованный `$state` содержит `parsedCsvData` с паролями).

**Рекомендация:** Использовать БД или временные файлы вне `DOCUMENT_ROOT` для хранения промежуточных данных импорта.

#### 2. Использование `$_REQUEST` без фильтрации метода

**Файл:** `lib/controller/apicontroller.php:202`
```php
if (!$data) {
    $data = $_REQUEST;
}
```

**Проблема:** `$_REQUEST` объединяет GET, POST и Cookie. При неудачном `json_decode` данных тела запроса, метод `getRequestParams()` падает на `$_REQUEST`, что может привести к обработке GET-параметров как данных write-операции.

**Рекомендация:** Для write-действий ограничиться только `$_POST`.

### 🟢 Низкий (Низкий) — 4

#### 1. Неявная утечка информации о существовании пользователей

**Файл:** `lib/model/rightmanager.php:940-944`
```php
$knownUsers = User::getUserListById(array_values($users));
foreach ($users as $id) {
    if (!isset($knownUsers[$id]) || ($knownUsers[$id]['ACTIVE'] ?? 'N') !== 'Y') {
        throw new \InvalidArgumentException('Unknown or inactive user');
    }
}
```

Сообщение об ошибке не раскрывает, какой именно пользователь неизвестен. Это хорошая практика.

#### 2. Отсутствие заголовков безопасности (CSP, X-Frame-Options)

Модуль не устанавливает `Content-Security-Policy` и `X-Frame-Options`. Это может быть проблемой при интеграции в окружения, где Bitrix не настроен на установку этих заголовков.

#### 3. Ограничение размера тела запроса жёстко задано

**Файл:** `lib/controller/apicontroller.php:191`
```php
$maxBodyBytes = 10485760; // 10 MB
```

Хорошая защита от DoS, но параметр не конфигурируемый.

#### 4. Использование `preg_match` без ограничения на длину строки

**Файл:** `lib/model/rightmanager.php:480`
```php
if (!preg_match_all('/(?=((?:^|[;{])s:4:"TEXT";s:(\d+):"))/', $value, $matches, PREG_OFFSET_CAPTURE)) {
```

Длина входной строки ограничена 2 МБ (строка 475), что предотвращает ReDoS-атаки через catasthropic backtracking.

## Битрикс-специфичные уязвимости

### ✅ Хорошая реализация:

1. **Использование `bitrix_sessid()` для CSRF-защиты** — `apicontroller.php:53` и фронтенд передаёт `csrf_token` в каждом запросе.

2. **Правильное использование `$APPLICATION->RestartBuffer()`** — вызывается перед выводом JSON, чтобы исключить примешивание HTML из шаблона.

3. **Использование Bitrix ORM** — `ItemTable` и `RightTable` используют `DataManager` от Bitrix, что защищает от SQL-инъекций.

4. **Использование `CIBlockElement::GetList` с фильтрацией** — корректное экранирование ID инфоблока, кастинг к `int`.

5. **События (event handlers)** — корректно зарегистрированы через `RegisterModuleDependences`:
   - `onIBlockSectionDelete` — очистка прав при удалении отдела
   - `onUserDelete` — передача владения записей администратору

6. **Authentication Filter** — `new Authentication()` в конфигурации действий API.

7. **IDOR защита** — при обновлении пароля проверяется, что запись принадлежит указанному разделу (строка 481):
   ```php
   if ((int)$entity['SECTION'] !== $params['SECTION']) {
       $this->sendJsonError('Нельзя изменить пароль в другом разделе', 403);
   }
   ```

### ⚠️ Потенциальные проблемы:

1. **Переопределение `$_SERVER['REQUEST_URI']`** — `install/frontend/api.php:36` изменяет глобальную переменную `$_SERVER['REQUEST_URI']`. Это может вызвать конфликты с другими модулями в многопоточном окружении.

2. **Использование глобальных констант** — `KEYRIGHTS_BASE_URL` определяется в `ckeyrights.php:49` без проверки на уже определённую константу. Хотя `defined()` используется, это потенциальный конфликт имён.

3. **Прямое включение `header.php` и `footer.php`** — `component.php:7,13` вручную подключает шаблон сайта, что может нарушить работу при изменении структуры шаблонов Bitrix.

4. **`$GLOBALS["reqCheck"]` в установщике** — `install/index.php:39` записывает данные в глобальную переменную, полагаясь на специфичное поведение Bitrix при включении файлов шагов установки.

## Рекомендации

### Обязательные (High):

1. **Ограничить API endpoint `api.php` только POST-запросами** и добавить проверку `check_bitrix_sessid()`:
   ```php
   // install/frontend/api.php - добавить после define'ов
   if ($_SERVER['REQUEST_METHOD'] !== 'POST') {
       http_response_code(405);
       die();
   }
   if (!check_bitrix_sessid()) {
       http_response_code(403);
       die();
   }
   ```

2. **Убрать `$_REQUEST` из `getRequestParams`** для write-действий — использовать только `$_POST`:
   ```php
   // apicontroller.php ~202
   if (in_array($route, self::WRITE_ACTIONS, true)) {
       $data = $_POST;
   }
   ```

### Рекомендуемые (Medium):

3. **Переместить временные данные импорта из `$_SESSION` в БД** для предотвращения утечек при компрометации сессий.

5. **Добавить логирование попыток несанкционированного доступа** для обнаружения атак.

### Косметические (Low):

7. Удалить пустые языковые константы в `lang/ru/messages.php:213` и `lang/ru/install/index.php:71`.
