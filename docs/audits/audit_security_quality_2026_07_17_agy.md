## Резюме

Был проведен независимый аудит безопасности, качества кодовой базы, наличия обфускации и «мертвого» кода для модуля хранения паролей `drdroid.keyrights` (версия 2.x).

В ходе аудита было подтверждено, что большая часть старых проблем и признаков обфускации (например, числовые имена переменных в инсталляторе, устаревшие библиотеки из сторонних CDN, небезопасные вызовы функции `unserialize()` на сырых данных) **были успешно вычищены и исправлены** в ходе предыдущих рефакторингов. Кодовая база приведена к современному стандарту Bitrix D7 (с использованием ORM `DataManager`, строгой фильтрации запросов на уровне REST-экшенов, корректной очистки буфера вывода).

Тем не менее, в модуле присутствует ряд уязвимостей безопасности среднего уровня, недочетов в покрытии тестами и архитектурных компромиссов.

## 2. Проблемы безопасности и уязвимости (Security Vulnerabilities)

### 2.1. Потенциальный фатал и обход валидации владельца (Средняя — SEC-01)
* **Файл:** [lib/model/user.php](file:///home/bitrix/ext_www/b24dev.hmara.xyz/dev_modules/drdroid.keyrights/lib/model/user.php#L97-L111)
* **Описание:** Метод `User::getUserById($id)` вызывает `$user = $userRes->Fetch()` и сразу обращается к нему как к массиву: `$user['UF_DEPARTMENT'] = ...`, не проверяя, была ли найдена запись. Если пользователь с переданным `ID` отсутствует или был удален, `$user` принимает значение `false`. В PHP 8.x попытка записи по ключу в булево значение приводит к фатальной ошибке (`Fatal Error`), либо возвращает некорректную структуру. Это позволяет обойти логику проверки существования владельца в методе `setOwnerAction()`.
* **Решение:** Добавить явную проверку в `User::getUserById()` сразу после Fetch:
  ```php
  if (!is_array($user)) {
      return false;
  }
  ```

### 2.2. Архитектурная уязвимость: Хранение и передача мастер-ключа в открытом виде (Средняя — SEC-02)
* **Файлы:** [classes/general/ckeyrights.php](file:///home/bitrix/ext_www/b24dev.hmara.xyz/dev_modules/drdroid.keyrights/classes/general/ckeyrights.php#L123-L186), `install/index.php`
* **Описание:** Для функционирования клиентского шифрования мастер-ключ `clientPassphrase` встраивается в HTML-код страницы в глобальную JavaScript-переменную `window.CONST.key`. По умолчанию ключи шифрования хранятся в открытом виде в таблице `b_option`.
* **Угроза:** Компрометация ключей при любой XSS-уязвимости на портале или при компрометации резервных копий базы данных.
* **Решение:** Рекомендуется настроить хранение ключей через механизм `bitrix-crypto` (модуль уже поддерживает такую интеграцию в `Crypt::loadSecret`), а на уровне архитектуры задокументировать модель угроз клиентского шифрования.

### 2.3. Внедрение BB-кодов в IM-оповещениях запроса доступа (Средняя — SEC-03)
* **Файл:** [lib/controller/apicontroller.php](file:///home/bitrix/ext_www/b24dev.hmara.xyz/dev_modules/drdroid.keyrights/lib/controller/apicontroller.php#L720-L751)
* **Описание:** В методе `requestAccessAction()` имя раздела `$section['NAME']` вставляется напрямую в BB-код сообщения: `... в разделе [url=...]' . $section['NAME'] . '[/url]`. Если злоумышленник назовет раздел с использованием закрывающего тега `[/url]`, он сможет внедрить произвольные внешние ссылки или изменить разметку уведомления, отправляемого владельцу.
* **Решение:** Экранировать BB-коды или очищать имя раздела с помощью функций валидации перед отправкой в модуль Веб-мессенджера (`im`).

### 2.4. Неатомарность удаления раздела и прав ACL (Средняя — SEC-04)
* **Файл:** [lib/controller/apicontroller.php](file:///home/bitrix/ext_www/b24dev.hmara.xyz/dev_modules/drdroid.keyrights/lib/controller/apicontroller.php#L368-L387)
* **Описание:** При удалении папки в `sectionRemoveAction()` сначала стираются права доступа из таблицы ACL (`deleteSectionRights()`), а затем удаляется сам инфоблок раздела Битрикса (`deleteSection()`). Если удаление раздела завершится ошибкой (например, из-за внешнего обработчика событий), раздел останется на портале, но станет общедоступным, так как его права уже стерты.
* **Решение:** Проверять успешность удаления раздела Битрикса перед очисткой таблиц ACL модуля.

## 3. Анализ качества кода и мертвых ресурсов (Code Quality & Dead Code)

### 3.1. Фиктивно проходящие интеграционные тесты (Средняя — QA-01)
* **Файл:** [tests/integration/http_access.php](file:///home/bitrix/ext_www/b24dev.hmara.xyz/dev_modules/drdroid.keyrights/tests/integration/http_access.php#L22-L29)
* **Описание:** Интеграционный тест при отсутствии секретов окружения выводит предупреждение `SKIP` и завершает работу с кодом `0`. В пайплайне CI сборка помечается зеленой, даже если тесты доступа к API фактически не запускались.
* **Решение:** На уровне CI принудительно устанавливать флаг `KEYRIGHTS_TEST_REQUIRED=1`, чтобы отсутствие секретов завершало джобу с кодом ошибки `1`.

### 3.2. Наличие `node_modules` в установочном каталоге (Средняя — QA-02)
* **Каталог:** `install/components/drdroid/keyrights/node_modules/`
* **Описание:** При сборке фронтенда локально генерируется папка `node_modules`. При ручной установке модуля через админку папка копируется целиком, засоряя рабочее окружение Bitrix тысячами мелких временных файлов NPM-зависимостей.
* **Решение:** Исключить `node_modules` из списка файлов для копирования в методе `installFiles()`.

### 3.3. Избыточные статические ресурсы в `static/` (Низкая — QA-03)
* **Каталог:** `install/components/drdroid/keyrights/static/`
* **Описание:** В сборке статики фронтенда содержатся неиспользуемые файлы демо-пользователей (`ivan.jpg`, `leha.jpg`), пустые таблицы стилей и legacy-части Bootstrap 3.1.1, которые раздувают размер модуля до 6+ МБ.
* **Решение:** Очистить дистрибутив от неиспользуемых графических и стилистических материалов.
