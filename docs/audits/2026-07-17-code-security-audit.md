# Аудит кода и безопасности `drdroid.keyrights`

Дата аудита: 17 июля 2026 года
Версия модуля: 2.0.1
Базовый коммит: `010bc8329a1e111f255f23e7f45ef08ff5b329f8`
Объект аудита: текущее рабочее дерево репозитория, включая незакоммиченные изменения

## Резюме

Первичный аудит обнаружил следующие проблемы:

| Критичность | Количество |
|---|---:|
| Критическая | 1 |
| Высокая | 3 |
| Средняя | 7 |
| Низкая | 5 |

Главные риски:

1. служебные таблицы прав переименованы с `sib_kr_*` в `dr_kr_*` без миграции;
2. раздел можно переместить в родительский раздел, на который у пользователя нет
   права записи;
3. административный установщик исполняет jQuery 1.7.2 со стороннего CDN;
4. любой авторизованный пользователь получает полный список разделов, владельцев
   и правил доступа, даже если не имеет доступа к соответствующим паролям.

При этом в API уже присутствуют полезные защитные меры: обязательная авторизация,
POST и CSRF для изменяющих запросов, серверная фильтрация списка паролей по ACL,
проверка IDOR при изменении записи, ограничение глобального удаления прав
администратором и аутентифицированное серверное шифрование AES-256-GCM.

импорта, TTL и привязка состояния импорта к пользователю, локальный jQuery,
предупреждение о чувствительности CSV и безопасное кодирование формы установки.
Поддержка Blowfish оставлена только как read-only путь для миграции старых данных
в AES-256-GCM.

## Область и методика

Проверены PHP-код, API-маршрутизация, модель прав, установщик и удаление модуля,
SQL для MySQL/PostgreSQL, ORM, шифрование, импорт/экспорт, клиентский JavaScript,
статические ресурсы, Composer/npm-зависимости, тесты и GitHub Actions.

Выполнены:

- ручной анализ потоков авторизации, ACL и изменения данных;
- проверка всех PHP-файлов через `php -l`;
- `composer validate`, `composer test`, `composer audit --locked`;
- `npm audit` для frontend lock-файла и проверка production-сборки;
- поиск опасных вызовов, внешних URL, секретов, отладочного и неиспользуемого
  кода;
- сравнение текущей схемы с `HEAD` и тегом `v2.0.0`.

Аудит не включал полноценный пентест работающего сайта, фаззинг, проверку
конфигурации веб-сервера и Bitrix, а также реальные интеграционные прогоны на
MySQL и PostgreSQL. Поэтому отсутствие находок вне перечисленных ниже не является
гарантией отсутствия уязвимостей.

## Найденные проблемы

### KR-01. Несовместимое переименование таблиц ACL без миграции — критическая

Текущий код использует `dr_kr_item` и `dr_kr_right`:

- `lib/orm/itemtable.php:18`;
- `lib/orm/righttable.php:14`;
- `install/index.php:80`, `install/index.php:465-507`;
- `install/db/mysql/install.sql:1-25`;
- `install/db/pgsql/install.sql:1-25`.

В базовом коммите и теге `v2.0.0` используются `sib_kr_item` и
`sib_kr_right`. Миграции данных или совместимого выбора существующей схемы нет.
После обновления установленного модуля ORM начнёт обращаться к новым таблицам,
которых нет, либо к пустым таблицам. Пароли останутся в инфоблоке, но владельцы и
ACL станут недоступны; возможны ошибки API и фактическая потеря настроек доступа.
Текущий тест лишь проверяет наличие нового имени в SQL и поэтому не обнаруживает
регрессию обновления.

Рекомендация: сохранить опубликованные имена `sib_kr_*` либо реализовать
идемпотентную транзакционную миграцию с копированием данных, проверкой количества
строк, индексов и внешних ключей. Добавить тест обновления с заполненной старой
схемой отдельно для MySQL и PostgreSQL. До этого не собирать релиз из текущего
рабочего дерева.

### KR-02. Обход проверки прав при перемещении раздела — высокая

`RightManager::checkRightsLevel()` возвращает уровень только первого элемента
переданного массива (`lib/model/rightmanager.php:599-602`). В
`sectionMoveAction()` ему передаются текущий и новый родитель, но фактически
проверяется только текущий раздел (`lib/controller/apicontroller.php:278-310`).
В `sectionSaveAction()` при изменении родителя проверяется право на изменяемый
раздел, но не право записи в новый родительский раздел
(`lib/controller/apicontroller.php:226-269`).

Пользователь с правом записи на раздел может переместить его в любую существующую
ветку, включая недоступную ему. Это нарушает границы ACL, меняет наследование и
может раскрыть данные другим пользователям либо нарушить доступность структуры.

Рекомендация: явно и независимо требовать `ACCESS_CAN_WRITE` на перемещаемый
раздел, старого родителя и нового родителя. Не использовать метод, возвращающий
право только первого элемента, для групповой проверки. Покрыть оба API-маршрута
интеграционными тестами разрешённых и запрещённых перемещений.

### KR-03. Сторонний устаревший JavaScript в административном установщике — высокая

`install/step1.php:7` загружает:

```html
<script src="//ajax.googleapis.com/ajax/libs/jquery/1.7.2/jquery.min.js"></script>
```

Скрипт исполняется в контексте привилегированной страницы администратора. URL
протокол-зависимый: при открытии админки по HTTP загрузка также выполняется без
TLS. Версия 1.7.2 устарела и не контролируется Composer/npm-аудитом проекта.
Компрометация канала или внешнего источника даст выполнение кода с правами
текущей административной сессии.

Рекомендация: удалить внешнюю загрузку и использовать штатный JavaScript Bitrix
либо несколько строк нативного JS. Не разрешать внешние скрипты на страницах
установки модуля.

### KR-04. Раскрытие структуры хранилища и полного ACL — высокая

`crypt/section/list` доступен любому авторизованному пользователю и без фильтрации
возвращает `getSectionsTree(true)` (`lib/controller/apicontroller.php:221-224`).
`getSectionsTree()` загружает все разделы и все строки прав
(`lib/model/rightmanager.php:83-126`, `lib/model/rightmanager.php:450-495`). В
ответ попадают названия и описания разделов, создатели, владельцы, пользователи и
группы ACL, блокировки и сроки действия — в том числе для веток, к которым у
вызывающего нет доступа.

Названия и описания папок парольного хранилища сами могут быть чувствительными.
Кроме того, ответ позволяет построить карту ответственных лиц и действующих
разрешений.

Рекомендация: рассчитывать доступ сервером и возвращать только разрешённые ветки
плюс минимально необходимые родительские узлы. Полный ACL выдавать лишь через
отдельный POST+CSRF-маршрут после проверки права управления конкретным объектом.

### KR-05. Владельцем можно назначить несуществующего пользователя — средняя

`User::getUserById()` не проверяет результат `Fetch()` перед обращением к нему как
к массиву (`lib/model/user.php:97-110`). В PHP 8.2 запись поля в `false`
автоматически превращает значение в непустой массив с предупреждением. Поэтому
проверка существования владельца в `setOwnerAction()`
(`lib/controller/apicontroller.php:638-666`) может принять несуществующий ID.

Результат — осиротевший объект ACL и потеря управления прежним владельцем; вернуть
доступ сможет администратор.

Рекомендация: немедленно возвращать `false`, если `Fetch()` не вернул массив;
проверять положительный ID и активность пользователя. Добавить тест назначения
несуществующего и удалённого пользователя.

### KR-06. Глобальные ключи хранятся рядом с данными и выдаются всем пользователям — средняя

Обе ключевые фразы сохраняются в `b_option` обычными строками
(`install/index.php:49-54`). Серверный ключ читается в
`lib/helper/crypt.php:23-28`, а единый клиентский ключ встраивается в страницу
каждого авторизованного пользователя (`classes/general/ckeyrights.php:103-145`,
`classes/general/ckeyrights.php:156-182`) и сохраняется в глобальном объекте
`window.__keyrightsClientKey` (`static/js/helpers/crypt.js:1-9`).

AES-256-GCM корректно защищает серверный конверт от подмены, но резервная копия БД
содержит и шифротексты, и оба ключа. Единый клиентский ключ невозможно отозвать у
одного пользователя без ротации всех записей; любой XSS в origin портала сразу
получает этот ключ.

Рекомендация: задокументировать модель угроз. Серверный master key хранить вне БД
или защищать ключом окружения. Для строгой изоляции применять отдельные data keys
на запись/раздел и оборачивать их ключами пользователей; предусмотреть версию
ключа, ротацию и процедуру восстановления.

### KR-07. CSV-формулы не нейтрализуются при экспорте истории — средняя

`static/js/helpers/csv.js:1-11` только экранирует кавычки и разделители. Значения,
начинающиеся с `=`, `+`, `-` или `@`, останутся формулами при открытии CSV в Excel
или другом табличном редакторе. В экспорт попадают имена пользователей и записей
(`lib/model/history.php:61-125`, `static/js/actions/root.js:302-319`), которые
могут содержать контролируемый текст.

Рекомендация: перед CSV-кодированием нейтрализовать формульные префиксы, например
добавлять апостроф, и добавить тесты для `=HYPERLINK(...)`, `+`, `-`, `@`, таба и
перевода строки.

### KR-08. Любой пользователь может выгрузить корпоративный справочник — средняя

Через `api/call-method` любой авторизованный пользователь может вызвать
`user.get` и `department.get` (`lib/controller/apicontroller.php:672-700`).
`User::getUserList()` возвращает всех пользователей, включая email, login,
должность, фото и подразделения (`lib/model/user.php:14-84`). Frontend делает эти
запросы при каждой загрузке (`static/js/actions/root.js:90-120`), даже если
пользователь не управляет правами.

Рекомендация: выдавать справочник только владельцу/пользователю с правом
управления конкретным ACL либо реализовать серверный поиск с минимальным набором
полей и ограничением количества результатов.

### KR-09. Установка и удаление раздела неатомарны — средняя

После успешного `InstallDB()` результаты `InstallIblocks()`, `installFiles()`,
rewrite и меню не проверяются, но модуль всё равно регистрируется
(`install/index.php:55-67`). Ключи создаются до этих операций. Ошибка оставляет
частично установленный модуль, таблицы, опции или файлы.

При удалении раздела сначала удаляются ACL, затем сам раздел, а результат удаления
раздела игнорируется (`lib/controller/apicontroller.php:319-336`). Если Bitrix не
удалит раздел, он останется без прежних прав.

Рекомендация: проверять каждую операцию, вести журнал выполненных шагов и делать
компенсирующий rollback. Для удаления сначала убедиться в успешном удалении
объекта либо предусмотреть восстанавливаемую двухфазную операцию.

### KR-10. Интеграционные тесты могут «успешно» ничего не проверить — средняя

`tests/integration/http_access.php:14-25` завершает работу с кодом 0, если нет
любого из секретов окружения. Workflow всегда запускает файл
(`.github/workflows/keyrights.yml:52-71`), поэтому зелёный job не доказывает, что
HTTP-тесты выполнялись. Тест содержит только два негативных сценария и не
проверяет перемещение разделов, раскрытие ACL, установку, обновление схемы,
MySQL/PostgreSQL или криптомиграцию.

Рекомендация: разделить явный `skip` и обязательный защищённый integration job;
для релиза завершать job ошибкой при отсутствии окружения. Поднять в CI временные
MySQL и PostgreSQL и тестировать установку плюс обновление с последней
опубликованной версии.

### KR-11. Запрос доступа допускает спам и подмену разметки сообщения — средняя

`api/request-access` доступен любому авторизованному пользователю. В
`lib/controller/apicontroller.php:722-760` нет ограничения частоты, защиты от
повторных запросов и проверки, что у отправителя действительно отсутствует
доступ. Название раздела без экранирования вставляется внутрь Bitrix BBCode
`[url]...[/url]` (`:756-757`), поэтому специально сформированное имя может
изменить текст уведомления или добавить ссылку.

Рекомендация: экранировать BBCode, проверять отсутствие доступа, ограничивать
частоту по пользователю/разделу и подавлять дубликаты на заданное время.

### KR-12. Ошибочные и неполные ACL принимаются неоднозначно — низкая

`saveEntityRights()` ограничивает только тип массива и число строк, но не требует
ровно одного субъекта `user`/`group`, существования субъекта и уникальности
(`lib/model/rightmanager.php:756-802`). Дубликаты прав одного пользователя
разрешены, а `_checkOneRight()` возвращает первое совпадение в порядке ID
(`lib/model/rightmanager.php:661-675`). Результат зависит от порядка строк.
Кроме того, `rightsSaveAction()` игнорирует `false` от `saveEntityRights()` и
возвращает успех (`lib/controller/apicontroller.php:583-610`).

Рекомендация: валидировать схему ACL, запретить пустые/двойные субъекты и
дубликаты, формально определить приоритет deny/allow и проверять результат модели.

### KR-13. Ошибки API часто возвращаются с HTTP 200 — низкая

`sendJsonError()` по умолчанию использует код 200
(`lib/controller/apicontroller.php:167-176`), и многие ошибки доступа вызываются
без явного 403. Это ухудшает мониторинг, работу reverse proxy/WAF и поведение
клиентов.

Рекомендация: применять 400 для неверного ввода, 401/403 для авторизации, 404 для
отсутствующего объекта, 409 для конфликта и 500 только для внутренних ошибок.

### KR-14. Ошибка сравнения дат временного доступа — низкая

`static/js/components/panel/right.js:14-18` сравнивает только день месяца через
`moment(d).date()`, а не полную дату. Например, будущая дата следующего месяца с
меньшим номером дня будет ошибочно отклонена. В
`static/js/components/panel/right-blocked.js:11-16` остался такой же неиспользуемый
метод, причём файл не импортирует `moment`.

Рекомендация: сравнивать `moment(d).startOf('day').isAfter(moment().startOf('day'))`
и удалить мёртвый метод из компонента заблокированного права.

### KR-15. Мусорные и устаревшие ресурсы увеличивают поверхность сопровождения — низкая

В дереве присутствуют отдельные скомпилированные CSS-части вместе с исходными
LESS, два пустых CSS-файла (`_ipad.css`, `_var.css`), неиспользуемые стили
Uploadifive, демо-изображения `cp.png`, `dribble.png`, `userpic/ivan.jpg`,
`userpic/leha.jpg`, старый Bootstrap 3.1.1 и крупные SVG-шрифты. Каталог `static`
занимает около 6,2 МБ, из них около 3,7 МБ — шрифты. В PHP остались неиспользуемые
`CEventLog` и поле `$ibs` в `lib/model/history.php:7,18,25`, а во frontend —
`console.log` в `static/js/components/panel/add-item.js:215`.

Рекомендация: составить белый список runtime-ресурсов, удалить демо/дубликаты и
неиспользуемые импорты, оставить один воспроизводимый CSS bundle и проверить
лицензии реально поставляемых шрифтов и библиотек.

### KR-16. GitHub Actions не закрепляет actions по commit SHA — низкая

Workflow использует плавающие major-теги, включая сторонний
`shivammathur/setup-php@v2` (`.github/workflows/keyrights.yml:19-20,42-43,59-60` и
`.github/workflows/release.yml:24-27,50,102`). Компрометация тега или upstream
повлияет на сборку релиза с `contents: write`.

Рекомендация: закрепить все actions на проверенных commit SHA, включить Dependabot
для GitHub Actions и по возможности разделить job сборки и job публикации с
минимальными permissions.

## Положительные наблюдения

- изменяющие API-маршруты централизованно требуют авторизацию, POST и CSRF;
- список паролей сначала фильтруется по ACL без расшифрования недоступных записей;
- обновление пароля проверяет соответствие ID и раздела;
- глобальное удаление прав ограничено администратором;
- серверный формат `v2` использует случайный nonce, AES-256-GCM, AAD и строгую
  проверку тега;
- legacy HTML-свойство разбирается ограниченным строковым парсером без вызова
  `unserialize()`;
- внешний URL записи разрешается только для `http`/`https`, а ссылка открывается
  с `noopener noreferrer`;
- JSON, встраиваемый в HTML, кодируется с `JSON_HEX_*`;
- явных `eval`, shell-вызовов, загрузки PHP-файлов пользователем и хардкодных
  production-секретов не найдено.

## Приоритет исправлений

1. Вернуть совместимые таблицы `sib_kr_*` или реализовать и протестировать
   миграцию; заблокировать релиз до проверки обновления на обеих БД.
2. Закрыть обход ACL при обоих вариантах перемещения раздела.
3. Удалить внешний jQuery из установщика.
4. Перенести фильтрацию дерева разделов и ACL на сервер.
5. Исправить проверку владельца и добавить негативные интеграционные тесты.
6. Затем устранить CSV injection, минимизировать справочник пользователей,
   сделать операции установки/удаления восстанавливаемыми и расширить CI.
